2022年8月、火曜日の朝。上海のオフィスで、私はGitHubのスターカウンターをじっと見ていた。

AFFiNEのCOOとして、私の仕事は「どうやってもっと多くの開発者にこのプロジェクトを知ってもらうか」だった。当時のスター数は3,000。目標は10,000。現実は遠かった——そう思っていた。

3年後の今、AFFiNEは**60,000スター**を超えている。

## Key Stats

| 指標 | 数値 |
|------|------|
| GitHub Stars | 60,000+ |
| Product Hunt #1 | 30回 |
| GitHub Trending登場 | 28回（5ヶ月） |
| コントリビューター | 300人以上 |
| MAU | 100,000+ |

これは運ではない。再現可能な戦略の結果だ。

## 戦略1：README を「製品ページ」として設計する

最初に気づいたのは、READMEが「説明書」になっていることの問題だ（技術者は説明書を書きたがる——それ自体は悪くない）。

でも訪問者が最初に見るのはREADMEだ。ここで離脱されたら、どれだけ良いコードを書いても意味がない。

**やったこと：**
- ファーストビューに「何ができるか」を1文で
- GIFアニメで動作を即座に見せる
- バッジ（Stars/Contributors/License）を目立つ場所に
- 「Quick Start」を3ステップ以内に収める

READMEを改善した翌週、スターの獲得ペースが2.3倍になった。

## 戦略2：「点火タイミング」を設計する

GitHub Trendingに載るには、**短期間での集中的なスター獲得**が必要だ。

私たちが使ったのは「点火」の概念。新機能リリースやブログ記事公開のタイミングに合わせて、複数のコミュニティに同時に投稿する。

**点火に使ったチャンネル：**
- Hacker News（Show HN）
- Reddit（r/programming, r/selfhosted）
- Product Hunt
- Twitter/X での開発者コミュニティ
- Discord サーバー

1回の点火で2,000〜5,000スターを獲得したことがある。

## 戦略3：コミュニティを「資産」として育てる

スターは一時的なものだ——でもコントリビューターは長期的な資産になる（これに気づくのに1年かかった）。

**Discordコミュニティの育て方：**
- 毎週「What are you building?」スレッド
- コントリビューターのPRを即座にマージ（24時間以内）
- バグ報告者には必ずお礼のコメント
- 月次ロードマップ公開

300人のアクティブコントリビューターが、口コミで毎月数百のスターを連れてきてくれる。

## 戦略4：言語別Trendingを狙う

GitHub Trendingには言語フィルターがある。

「All Languages」より「Python」「TypeScript」などの言語別の方が競争が少ない。AFFiNEはTypeScriptで書かれているので、TypeScript Trendingに狙いを絞ることで、より長くTrendingに居続けられた。

## 戦略5：外部メディアへの露出を計画的に

スター数が増えると、メディアが取り上げてくれやすくなる——しかし待っていてはいけない。

**積極的にやったこと：**
- 有名な「Awesome系」リストへのPR
- 技術ブログへの寄稿
- YouTuberへのデモ提供
- 製品比較サイトへの登録

これらすべてが、バックリンクとしてスター獲得に貢献する。

## まとめ：スターは「結果」ではなく「プロセス」

スターを増やすことを目的にすると、必ず間違った方向に進む——私はそう思っている。

本当の目的は「開発者に価値を届けること」。スターはその結果として自然についてくる。

私たちの全戦略とツールは [Gingiris Growth Tools](https://gingiris.github.io/growth-tools/) で公開している。
